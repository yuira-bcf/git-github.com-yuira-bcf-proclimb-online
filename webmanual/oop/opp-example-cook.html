オブジェクト指向を「料理のレトルト」で例える

オブジェクト指向、難しく聞こえるけど、レトルトで考えると胃袋にも頭にも優しい。

クラスとオブジェクト

クラス（Class）＝レトルトの「設計図」

例：ビーフカレー（中辛）という商品仕様。具、辛さ、内容量、加熱方法などを決めた“型”。

オブジェクト（Object）＝実際のレトルトパウチ1個

スーパーで買ったこの1袋。同じ設計図から実体がいくつも作れる。

例：

  • クラス：レトルトカレー
  • オブジェクト：昨日買ったパウチA、今日買ったパウチB

プロパティ（属性）とメソッド（振る舞い）

属性（プロパティ）＝持っている情報

  • 辛さ：spice_level = 3
  • 内容量：grams = 180
  • 賞味期限：best_before = 2026-05-01
  • 原材料：ingredients = [...]

メソッド（メンバ関数）＝できること

  • heat()：温める
  • open()：開ける
  • serve()：盛り付ける
  • eat()：食べる（※これはあなたのメソッド）

カプセル化（Encapsulation）

「中身の製造工程は気にするな」がカプセル化。

レトルトは工場の工程を知らなくても食える。同様に、内部データや処理を隠して、外部からは決まった操作だけできるようにする。

あなたは 温める と 開ける を呼べばOK。
工場の工程を直接いじり始めたら、それはもう料理じゃなくて事故。

継承（Inheritance）

「レトルト食品」という親カテゴリを作って共通部分をまとめるイメージ。

  • 親クラス：RetortMeal（共通：heat(), open(), serve() など）
  • 子クラス：Curry, Gyudon, MapoTofu（違いだけ追加・変更）

ポリモーフィズム（Polymorphism）

「とにかく heat() すれば正しく温まる」が最大の旨味。

カレーでも牛丼でも麻婆でも、同じ命令（heat()）で、それぞれの流儀で動く。

  • カレー：湯せん5分
  • 牛丼：レンジ2分
  • 麻婆：湯せん推奨…など

コンポジション（Composition）

セット商品＝部品の組み合わせ。

「カレーライスセット」は カレーパウチ + ご飯パック + 福神漬け みたいに、オブジェクトを部品として組み合わせて別のオブジェクトを作る発想。

継承で全部表そうとすると、食品が哲学になります（おいしくない）。

ざっくりコード（Python風）

from abc import ABC, abstractmethod

class RetortMeal(ABC):
    def __init__(self, name, grams):
        self.name = name
        self.grams = grams

    @abstractmethod
    def heat(self):
        pass

    def open(self):
        return f"{self.name} を開封した"

class Curry(RetortMeal):
    def heat(self):
        return f"{self.name} は湯せん5分で温める"

class Gyudon(RetortMeal):
    def heat(self):
        return f"{self.name} はレンジ2分で温める"

meals = [Curry("ビーフカレー", 180), Gyudon("牛丼の具", 160)]

for m in meals:
    print(m.heat())   # 同じ heat() でも結果が違う（ポリモーフィズム）

まとめ（レトルトで覚える一言）

  • クラス＝商品設計
  • オブジェクト＝実際の1袋
  • カプセル化＝中身は触るな、温めろ
  • 継承＝レトルト食品という親子関係
  • ポリモーフィズム＝同じ「温める」で各自勝手に正しく動く
  • コンポジション＝セットは部品の組み合わせ

オブジェクト指向を誤用すると「インスタント味噌汁に設計書200ページ」みたいな地獄が始まるので、レトルト同様“必要な分だけ”がちょうどいい。
