1. 概要（何者か）

コンストラクターとは、オブジェクトを生成した直後に自動で実行される特別なメソッドです。 主な役割は、オブジェクトを正しい初期状態にすることです。

ポイント： Javaでは new によってオブジェクトを生成すると、対応するコンストラクターが必ず1回だけ呼び出されます。

2. 定義（文法とルール）

コンストラクターの基本例です。

class User {
    String name;
    int age;

    User(String name, int age) {
        this.name = name;
        this.age = age;
    }
}

基本ルール

名前
クラス名と同じ
戻り値
書かない（void も書けません）
呼び出し
new と同時に自動実行
回数
原則として生成1回につき1回
補足：コンストラクターを1つも定義しない場合、Javaは「引数なしコンストラクター」を自動で用意します。 ただし、別のコンストラクターを定義した時点で、自動生成はされなくなります。

3. 必要性（なぜ必要か）

コンストラクターが必要な理由は、オブジェクトを「使える状態」で完成させるためです。

理由1：必須情報の設定漏れを減らす

例えば「名前は必須」のように、作成時点で必要な情報を引数として受け取る設計にすると、 必須情報の不足をコンパイル時に気づきやすくなります。

理由2：初期化ルールを一か所に集約できる

初期値の決め方や、入力値の扱い（例：空文字を許可するか）などをコンストラクターにまとめると、 実装者ごとの差が出にくくなります。

4. 使い所（実務の代表例）

① 必須フィールドの初期化

class Order {
    private final int price;

    Order(int price) {
        this.price = price;
    }
}

② 初期状態の保証（初期値を統一）

class Account {
    private int balance;

    Account() {
        this.balance = 0;
    }
}

③ 依存関係の受け取り（コンストラクター注入）

class Service {
    private final Repository repo;

    Service(Repository repo) {
        this.repo = repo;
    }
}
ポイント： 「何が必要で」「何を受け取って」作られるのかが明確になり、設計意図が読み取りやすくなります。

5. メモリとの関連（ヒープとスタック）

次の1行を例に、内部で起きる流れを整理します。重要 

User user = new User("Taro", 20);

内部で起きる流れ

  1. ヒープ領域に User オブジェクト分のメモリが確保されます。
  2. フィールドは一度、デフォルト値で初期化されます（例：int は 0、参照型は null）。
  3. コンストラクターが実行され、意図した値（"Taro" と 20）が設定されます。
  4. スタック領域の変数 user に、ヒープ上オブジェクトの参照（アドレス情報）が代入されます。
まとめ： コンストラクターは「メモリ確保の後」に動作し、オブジェクトを完成状態に仕上げる工程です。

6. 引数なしコンストラクターは定義するべきか

原則として、引数なしコンストラクターは定義しないほうが望ましいです。 理由は、必須項目が未設定のままオブジェクトを生成できてしまうためです。

例外的に定義する代表例

  • フレームワークが内部で生成する必要がある場合（例：ORM、リフレクション等）
  • 段階的に設定する設計であり、その意図が明確な場合
注意： 引数なしコンストラクターを用意する場合は、なぜ必要なのかを設計書に明記し、 業務コードでの利用範囲を明確にすることが大切です。

7. チーム開発での設計指針

チーム開発では、設計段階で次の点まで決めておくと、記述漏れが起きにくくなります。

設計で決める項目 目的
必須項目／任意項目の区別 生成時に必要な情報を明確にする
使用するコンストラクター一覧 生成方法を統一し、実装差を減らす
初期値・入力ルール 初期化の方針を一貫させる
フィールドのアクセス制御（例：private, final） 不用意な変更を防ぎ、意図した状態を保つ

8. まとめ

  • コンストラクターは、オブジェクト生成直後に自動で実行され、初期状態を整えます。
  • 必須項目をコンストラクター引数として定義すると、設定漏れを減らせます。
  • コンストラクターは、メモリ確保（ヒープ）後の「仕上げ工程」として働きます。
  • 引数なしコンストラクターは原則として定義せず、必要な場合は理由を明確にします。
  • チーム開発では、生成条件・初期化ルール・コンストラクター仕様を設計書に明記すると効果的です。
教科書の要点： 「オブジェクトは、生成された瞬間に正しい状態であることが望ましい」 — この考え方を支える中心的な仕組みがコンストラクターです。
