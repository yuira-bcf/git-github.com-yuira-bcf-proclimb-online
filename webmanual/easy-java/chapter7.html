オブジェクト指向

1. オブジェクト指向誕生の歴史

初期のプログラムは、上から下へ処理を順番に書く 手続き型プログラミングが主流でした。 規模が小さいうちは問題ありませんでしたが、 プログラムが大きくなるにつれて次のような問題が発生しました。

  • 処理の流れが複雑になり、全体像が分からなくなる
  • 1か所の修正が、別の場所に影響しやすい
  • 人が増えると、管理や分業が難しい

こうした問題を解決するために、 「現実世界の考え方をプログラムに持ち込もう」 という発想から生まれたのが オブジェクト指向です。

オブジェクト指向は「大規模・長期開発」に対応するために生まれた考え方

2. オブジェクト指向とは

オブジェクト指向とは、 プログラムを 「モノ（オブジェクト）」の集まり として考える設計手法です。

それぞれのオブジェクトは、

  • データ（状態）
  • 処理（振る舞い）

をひとまとまりとして持ちます。

オブジェクト ＝ データ ＋ 処理

例えば「人」をプログラムで表す場合、 名前や年齢といった情報（データ）と、 話す・動くといった動作（処理）を 1つのオブジェクトとして扱います。

3. オブジェクト指向の定義と効果

3-1. オブジェクト指向の定義

オブジェクト指向は、次のように定義できます。

プログラムを、役割を持ったオブジェクト同士の
やり取りとして設計・実装する考え方

Java では、この考え方を クラスとインスタンス という仕組みで表現します。

3-2. 効果

  • 処理とデータの関係が明確になる
  • 役割分担がはっきりする
  • 変更に強い構造になる

4. オブジェクト指向のメリット

4-1. 可読性が高くなる

オブジェクトごとに役割が分かれているため、 「どこに何が書いてあるか」が分かりやすくなります。

4-2. 保守・修正がしやすい

変更が必要な場合でも、 関連するオブジェクトだけを修正すればよく、 影響範囲を限定できます。

4-3. 再利用しやすい

一度作ったクラスは、 他のプログラムや機能でも再利用できます。

4-4. 分業しやすい

オブジェクトごとに作業を分担できるため、 チーム開発に向いています。

オブジェクト指向は「作る人が増えても破綻しにくい」設計手法

まとめ

・オブジェクト指向は、プログラムの大規模化に対応するために生まれた。
・プログラムを「オブジェクトの集まり」として考える設計手法である。
・データと処理をひとまとまりにすることで、構造が分かりやすくなる。
・可読性、保守性、再利用性、分業のしやすさが大きなメリットである。

オブジェクト指向の全体像と本質

1. オブジェクト指向と現実世界

オブジェクト指向では、プログラムを 「現実世界のモノや役割」に近い形で表現します。 たとえば、学校の仕組みを考えると、 「生徒」「先生」「授業」「教室」などの登場人物（モノ）が存在し、 それぞれが役割を持って動いています。

現実世界の「モノ」と「役割」を、プログラム上のオブジェクトとして表す

重要なのは、オブジェクト指向は「現実のコピー」ではなく、 プログラムとして必要な範囲だけを切り出して表現する という点です。

2. 開発時に作るクラス

Java の開発では、まず クラスを作成します。 クラスは「設計図」であり、 どのようなデータ（状態）と処理（振る舞い）を持つか を定義します。

クラス = オブジェクトの設計図（開発時に作る）

例として「生徒」を表すクラスを考えます。 

public class Student {
    String name;
    int score;

    void show() {
        System.out.println(name + " : " + score);
    }
}

ポイント：
クラスの中には「情報（フィールド）」と「動作（メソッド）」をまとめて書きます。 これにより「データと処理の関係」が自然に結び付きます。

3. 実行時に動くオブジェクト

クラスは設計図なので、作っただけでは動きません。 実行時には new を使ってオブジェクト（インスタンス）を作成し、 それを動かしてプログラムを進めます。

オブジェクト（インスタンス）= クラスから作られる実体（実行時に動く）

例として、先ほどの Student クラスから オブジェクトを作成して使用します。 

public class Main {
    public static void main(String[] args) {

        Student s1 = new Student();
        s1.name = "田中";
        s1.score = 80;

        Student s2 = new Student();
        s2.name = "佐藤";
        s2.score = 90;

        s1.show();
        s2.show();
    }
}

ここで重要な点：

  • Student クラスは1つでも、オブジェクトは何個でも作れる
  • それぞれのオブジェクトは、独自のデータ（name/score）を持つ
  • 同じメソッド（show）でも、表示内容はオブジェクトごとに変わる

4. オブジェクト指向における開発者の役割

オブジェクト指向で開発者が行うべきことは、 単にクラスを書くだけではありません。 最も重要なのは、次の判断です。

「どんなオブジェクトが必要か」
「それぞれにどんな責任（役割）を持たせるか」
を決めることが開発者の役割

つまり、開発者はプログラムの世界の「登場人物」と「役割分担」を設計します。

設計時の主な観点：

  • 現実世界の要素のうち、プログラムに必要なものだけを選ぶ
  • 1つのクラスに役割を詰め込みすぎない（責任を分ける）
  • 変更が起きそうな部分を分離して、影響範囲を小さくする

5. オブジェクト思考の本質

初学者が「オブジェクト指向」を難しく感じる原因は、 クラスや new の書き方よりも、 考え方（設計の視点）が新しいためです。

オブジェクト思考の本質は、次の1点に集約できます。

変化しやすい現実に対応するために、
責任（役割）を分けたオブジェクト同士のやり取りとして設計する

さらに言い換えると、

  • 「この処理は誰の責任か？」を常に考える
  • 「情報を持っているオブジェクトが、その情報に関する処理も担当する」
  • 「Main に全部書かない。役割を分散する」

この考え方が身につくと、プログラムは次のように変わります。

オブジェクト指向が弱い設計 オブジェクト思考の設計
Main に処理とデータが集中する 役割ごとにクラスが分かれ、見通しが良い
変更に弱く、影響範囲が読めない 変更が局所化され、修正がしやすい
同じ処理が重複しやすい 再利用できる部品として整理される

まとめ

・オブジェクト指向は、現実世界の「モノ」と「役割」を参考にプログラムを設計する考え方である。
・開発時にクラス（設計図）を作り、実行時にオブジェクト（実体）が動く。
・開発者の役割は「必要なオブジェクト」と「責任（役割分担）」を設計することである。
・オブジェクト思考の本質は、責任を分けたオブジェクト同士のやり取りとしてプログラムを組み立てる点にある。
